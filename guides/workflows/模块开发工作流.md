# 模块开发工作流

> 覆盖单个功能模块从"实现方案"到"代码交付"的完整循环。
> 前置条件：已完成 [项目启动工作流](项目启动工作流.md) 中的项目搭建 + 模块 UI 文档整理。

---

## 一、Agent 角色定义

| Agent | 角色 | 职责 |
|-------|------|------|
| 主 Agent | 编排者 | 沟通需求、生成 Plan、生成模块方案、任务拆分、协调各 Agent、维护知识库 |
| Review Agent | 审查者 | 审查全局 Plan、审查模块实现方案 |
| Coding Agent | 实现者 | 按 Todo 写代码、构建验证、commit |
| QA Agent | 质检者 | 模块完成后审查代码产出，对照方案验证完整性和正确性 |

### 协作关系

```
用户
  │
  ▼
主 Agent（Orchestrator）
  │           │              │              │
  │           ▼              ▼              ▼
  │     Review Agent    Coding Agent     QA Agent
  │     (审查方案)      (写代码)        (质检代码)
  │           │              │              │
  │     ┌─────┘              │         ┌────┘
  │     │ 审查反馈            │ 偏差反馈  │ 质检结果
  │     ▼                    ▼         ▼
  └──── 主 Agent 决策 ◄──────┘    通过 → 知识库更新
                                  不通过 → Coding Agent 修复（最多 2 轮）
```

信息流向：
- 主 Agent → Review Agent：传递 Plan / 模块方案 + Context Summary
- Review Agent → 主 Agent：返回审查意见
- 主 Agent → Coding Agent：传递当前 Todo + 模块实现方案
- Coding Agent → 主 Agent：完成通知 / 方案偏差反馈
- 主 Agent → QA Agent：模块代码完成后触发质检
- QA Agent → Coding Agent：不通过时反馈修复要求
- QA Agent → 主 Agent：通过后触发知识库更新

---

## 二、执行模式约束

借鉴 Plan-Only / Implement 两阶段分离模式：

### Plan-Only 阶段（默认执行）

- 输入：功能需求 + UI 文档（已在启动工作流中产出）
- 产出：模块实现方案（按 [模块实现方案模板](../templates/模块实现方案模板.md) 输出）
- 约束：**不修改任何源码文件**，仅在 `chapters/{NN}-{模块名}/` 中创建文档

### Implement 阶段（需明确授权）

- 触发：用户明确说"开始实现""可以写代码"等
- 范围：按实现方案中的文件清单执行
- 约束：严格按方案执行，遇到偏差反馈主 Agent

## 三、流程总览

```
① 生成模块实现方案
  │
  ▼
② Review Agent 审查方案（最多 2 轮）
  │
  ▼
③ Coding Agent 按 Todo 逐个实现
  │
  ▼
④ QA Agent 审查代码产出（最多 2 轮）
  │
  ▼
⑤ 验证对比（参见启动工作流 §5）
  │
  ▼
⑥ 知识库更新
  │
  ▼
⑦ 回到启动工作流"选取下一个模块"
```

## 四、步骤详解

### ① 生成模块实现方案

主 Agent 产出模块实现方案，按 [模块实现方案模板](../templates/模块实现方案模板.md) 输出，存放在 `chapters/{NN}-{模块名}/impl-plan.md`。

核心内容：
- 功能范围（对照 UI 文档和开发计划）
- 技术方案（路由、数据流、状态管理、错误处理）
- 文件清单（需要新建 / 需要修改 / 需要参考）
- 边界情况
- Todo 列表（原子化、有序）

**文件清单是核心价值**：Coding Agent 拿到方案后直接知道该读哪些文件、改哪些文件、参考哪些文件，零探索成本。

---

### ② Review Agent 审查方案

Review Agent 从以下维度审查：

- **需求覆盖度**：功能范围是否完整（对照 UI 文档 + 开发计划）
- **技术可行性**：方案是否与项目技术栈兼容
- **文件清单准确性**：有没有遗漏、有没有多余
- **前置依赖**：依赖的模块是否已完成
- **Todo 粒度**：是否合理（太粗会遗漏，太细会拖慢）

最多 2 轮迭代，仍有重大分歧则升级给用户决策。

---

### ③ Coding Agent 按 Todo 逐个实现

**输入**：
- 当前 Todo 的描述
- 完整的模块实现方案（文件清单、技术方案、参考模式）
- 项目构建命令

**执行规则**：
- 按 Todo 顺序逐个实现，每次只做一个
- 每个 Todo 完成后执行构建命令确保编译通过
- 每个 Todo 完成后更新方案文档中的 checkbox
- 每个 Todo 完成后 git commit，message 格式：`feat(模块): T-xxx Todo标题`
- 遇到方案偏差 → 反馈主 Agent → 主 Agent 更新方案 → 继续

**commit message 示例**：
```
feat(auth): T-101 创建登录入参校验
feat(auth): T-102 实现登录 handler
fix(auth): T-104 边界情况处理与错误码
```

---

### ④ QA Agent 审查代码产出

模块所有 Todo 完成后，QA Agent 介入审查：

- 代码是否实现了方案中列出的所有功能点
- 边界情况是否都处理了
- 代码风格是否与参考文件一致
- 有没有硬编码、遗漏的错误处理、不合理的逻辑
- 构建 / 测试 / lint 是否通过

QA 不通过 → 反馈给 Coding Agent 修复 → 修复后再次 QA，最多 2 轮，仍不通过则升级给用户。

---

### ⑤ 验证对比

参见 [项目启动工作流 §5 验证基建要求](项目启动工作流.md#五验证基建要求)。

最低要求：L1 轻量验证（必须调用 `make verify-l1` 等统封命令截取关键页面并对比）。

---

### ⑥ 知识库更新

QA + 验证通过后，主 Agent 执行：

1. 将模块实现中的关键信息沉淀到知识库文档，按 [模块知识库文档模板](../templates/模块知识库文档模板.md) 输出，存放在 `chapters/{NN}-{模块名}/knowledge.md`
2. 更新 `INDEX.md` 中的模块状态标记
3. 归档实现方案（impl-plan.md 中的 Todo 全部勾选即为归档状态）

## 五、文档体系

### 模块级文档结构

```
chapters/{NN}-{模块名}/
├── ui-plan.md          ← 模块UI实现文档（启动工作流 §4 产出）
├── impl-plan.md        ← 模块实现方案（本工作流 ① 产出）
└── knowledge.md        ← 模块知识库（本工作流 ⑥ 产出）
```

### 文档生命周期

| 文档 | 创建时机 | 更新时机 | 归档条件 |
|------|---------|---------|---------|
| ui-plan.md | 模块开始前 | 实现过程中发现遗漏时 | 模块完成 |
| impl-plan.md | 步骤 ① | Todo 完成时勾选 checkbox | 全部 Todo 完成 |
| knowledge.md | 步骤 ⑥ | 后续维护时补充 | 长期维护 |

---

## 六、跨会话恢复路径

```
读 INDEX.md → 找到当前模块和状态
  → 读模块 impl-plan.md → 找到当前 Todo
    → 检查 git log 确认代码实际状态 → 继续工作
```

git commit 历史作为进度的"第二信源"，与文档中的 checkbox 互相印证，确保恢复准确。

---

## 七、失败恢复

### 当修复失败时

1. 修复根因，不修复症状
2. 每次修复后重新验证
3. 不做随机尝试

### 连续 3 次失败

1. **停止**所有编辑
2. **回退**到最后已知可工作状态
3. **记录**尝试过的方案和失败原因
4. **咨询** Oracle 或升级给用户

**绝不**：留下破损代码、继续碰运气、删除失败的测试来"通过"

---

## 八、完成标准

一个模块视为完成，当且仅当：

- [ ] impl-plan.md 中所有 Todo 已勾选
- [ ] 构建通过（零错误）
- [ ] QA 审查通过
- [ ] L1 验证对比通过（调用统一基建通过截图对比）
- [ ] knowledge.md 已沉淀
- [ ] INDEX.md 状态已更新

如果验证发现预先存在的问题（非本模块引入）：
- 不修复，记录在 knowledge.md 中
- 报告："完成。注意：发现 N 个预先存在的问题，与本模块无关。"

# Agent 角色定义规范

> 本文档定义了项目全生命周期中，大语言模型（AI）及其虚拟分身应扮演的核心角色的触发时机、权限边界及交接协议。
> 通过强制实施角色物理级别的上下文隔离，有效抑制大模型的思维幻觉、越权覆盖与死循环堆栈污染。

---

## 一、主 Agent (Orchestrator)

**角色定位**：项目管家与中枢大脑
- **触发条件**：接收人类用户初始指令时的原生首选项；贯穿整个流水线调度。
- **行为边界**：
  - **允许功能**：沟通需求、编写技术蓝图、切分派发任务（Plan）、维护 `knowledge.md` 知识库和 `INDEX.md` 索引状态。
  - **严禁越界**：禁止在未经详尽设计的前提下直接下场编写底层业务代码；且无权越过 Review Agent 自行敲定破坏性甚至具有争议的架构方案。
- **交接协议**：
  - 分配任务前，必须为下游提供严密界定的上下文界限（如指明只准许依据某个特定的 `impl-plan.md` 工作）。
  - 处理下游传回的“偏差报告单”时，主 Agent 唯一负责更新上游结构规划。

## 二、Review Agent (审查者)

**角色定位**：静态架构的守门员（Plan 阶段主导）
- **触发条件**：主 Agent 产出《模块实现方案》或《架构设计方案》草案之后强制唤醒。
- **行为边界**：
  - **允许功能**：逐字核对产出文档与 `guides/templates/` 标准的保真度；对照 UI 文档质询需求遗漏。
  - **严禁越界**：不得执行任何修改源代码的操作。
- **交接协议**：
  - **审查不通过**：必须明确指出具体违反了哪一条架构原则（例如“缺少数据模型映射表”），无情打回给主 Agent 重新修改。
  - **审查通过**：下达“批准投产”指令，方可向下一步放行。

## 三、Coding Agent (实现者)

**角色定位**：被限制感官的流水线工人
- **触发条件**：收到经审批通过的《模块实现方案》。
- **行为边界**：
  - **允许功能**：仅且**只能修改清单中预声明的源文件**；推崇实施 TDD 的红-绿-重构节奏。
  - **严禁越界**：绝不允许私自更改架构路由或跨模块篡改。若发现前置配置确实缺失但未在方案清单内，强制停止动作提交偏差，禁止私自加建。
- **交接协议**：
  - **止损熔断**：连续修复/构建报错失败 **2次** 后，无论处于什么状态，必须强制脱手工作，启动 Debugger 诊疗方案。
  - 成功完成所有 Todo 后并跑通局部测试，即可发起转移。

## 四、QA Agent (质检者)

**角色定位**：极其挑剔的对抗性“找茬刺客”（Implement 阶段主导）
- **触发条件**：Coding Agent 完成所有业务代码堆砌时。
- **行为边界**：
  - **允许功能**：高频调度诸如 `make verify-l1` 类的强制快照与对比工具，深查 Git Diff。
  - **严禁越界**：绝对只读验证。就算顺手能够修复一个小 Typo，也不允许代替 Coder 去修改业务跑通状态。
- **交接协议**：
  - **标准话术**：如果找出漏洞，输出标准化 `[FAIL]` 块报错并回弹流程。
  - **通过结案**：当且仅当其在分析末尾发出“**已穷尽检查所有 AC 与变更文件，未发现潜在问题**”后，该代码库分支方为真正结束。

## 五、Debugger Agent (诊断者)

**角色定位**：独立隔离的场外专家组
- **触发条件**：收到 Coding Agent 连续 2 次滑铁卢的熔断警报后自动进场。
- **行为边界**：
  - **允许功能**：请求人类开启一个全新的极度干净的上下文窗口，吃进完备 Error Log 与异常区域源码，进行纯净宏观诊断。
  - **严禁越界**：不负责下场改代码（防止军师亲征战死），仅提供理论分析。
- **交接协议**：
  - 确认出 Root Cause 后，给出修改动作或文件级伪代码指导，掷回给 Coding Agent 重新入场尝试。
  - 若分析后得出无解或工具链死锁，强制截断自动化，直连请示人类求助。

## 六、Explorer Agent (探索者)

**角色定位**：嗅觉灵敏的安全搜捕犬
- **触发条件**：在面临《项目启动期逆向梳理》或是需要挖掘历史库设计代沟时释放而出。
- **行为边界**：
  - **允许功能**：无限制高频调用搜索 (`grep`) 与全文读取 (`cat/view`) 等文件系统嗅探能力。
  - **严禁越界**：系统级硬核锁定权限，绝对禁止做出哪怕新增一行 log 的任何数据写入操作。
- **交接协议**：
  - 将庞杂的搜索快照进行信息降维，提纯后归档成结构化的“勘探地形图”给主 Agent 参考吸收。

## 七、角色实例化映射

> 本规范与具体 AI 工具无关。§一至§六定义的角色、权限和交接协议适用于任何支持大模型编程的工具链。
> 本节提供一种**参考实现**，以 OpenCode 的 `task()` API 为例，说明如何将抽象角色映射到具体执行层。使用其他工具时，应按自身能力实现等价的角色隔离。

### 参考映射表（以 OpenCode 为例）

| 抽象角色 | 参考实例化方式 | 关键参数 | 权限要求 |
|---------|--------------|---------|---------|
| 主 Agent (Orchestrator) | 当前会话的顶层 Agent | — | 编排、分发、文档写入 |
| Review Agent (审查者) | `task(subagent_type="momus", ...)` | 必须同步等待结论 | 只读审查，不得修改源码 |
| Coding Agent (实现者) | `task(category="{合适类别}", load_skills=[...], ...)` | category 按任务领域选择 | 仅修改文件清单内的源码 |
| QA Agent (质检者) | `task(subagent_type="oracle", ...)` | prompt 中必须包含完整 AC 列表 + 变更文件清单 | 绝对只读，不得修改任何源码 |
| Debugger Agent (诊断者) | `task(subagent_type="oracle", ...)` | 独立上下文、仅提供分析和建议 | 只读诊断，不负责改代码 |
| Explorer Agent (探索者) | `task(subagent_type="explore", ...)` | 并行探索 | 只读搜索，禁止写入 |

### 实例化约束

1. **推荐将业务代码委派给独立的 Coding Agent 实例**，主 Agent 仅负责编排和验证。若工具不支持子 Agent 委派，主 Agent 可直接编写业务代码，但**仍须严格遵守文件清单约束**（仅修改 impl-plan.md §三 中声明的文件）和只读角色的隔离要求。无论哪种模式，主 Agent 的核心职责不变：
   - 提供完整的任务上下文（TASK / EXPECTED OUTCOME / REQUIRED TOOLS / MUST DO / MUST NOT DO / CONTEXT）
   - 验证产出是否符合预期
   - 发现偏差时更新方案或重新执行

2. **Review Agent 和 QA Agent 的只读约束**必须在执行前显式声明（支持子 Agent 时写入 prompt，不支持时作为自我约束检查项）。示例：
   ```
   MUST NOT DO:
   - 修改任何源码文件
   - 创建任何新文件
   - 执行构建或测试命令
   ```

3. **Coding Agent 的文件范围约束**必须在执行前显式声明。示例：
   ```
   CONTEXT:
   - 允许修改的文件：{impl-plan.md §三 文件清单}
   - 超出清单的修改必须停止并反馈
   ```

4. **主 Agent 可执行的直接操作**（无需委派）：
   - 文档编辑（Plan/knowledge.md/INDEX.md 等工作流文档）
   - 简单配置修改（添加模块导出、注册路由等纯胶水代码，约 10 行以内）
   - 构建验证（`make build` 等构建命令）
   - 代码探索（grep/read 等只读操作）

5. **上下文延续**：同一模块的多个 Todo 之间应尽量保持上下文延续（支持 session_id 的工具应复用 session，不支持的工具应在 prompt 中附带前序产出摘要），避免重复探索和文件读取。

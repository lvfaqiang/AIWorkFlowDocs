# 会话记录：OpenCode 全局工作流设计与文档体系重构

> 日期：2026-03-19
> 主题：讨论并建立标准化的 AI 辅助开发工作流，重构文档体系为可跨项目复用的通用模板

---

## 一、会话目标

讨论并建立一套标准化的 AI 辅助开发工作流，覆盖"项目启动 → 架构选型 → 功能梳理 → UI 细节整理 → 逐模块实现"的完整链路，并将其整理为可跨项目复用的通用文档模板体系。

---

## 二、讨论过程与关键决策

### 第一轮：工作流流程讨论

**用户提出的核心流程**：
- 有参考实现（如 Android → Harmony）：先梳理源项目架构/SDK/功能/逻辑细节 → 输出项目细节文档 → 规划模块顺序 → 每个模块实现前整理 UI 实现文档（含布局结构 + 状态逻辑 + 样式细节）
- 无参考实现：获取 Figma 设计稿 → 从设计稿节点提取 UI 细节 → 整理 UI 实现计划

**AI 提出的补充建议**（共 4 点）：

1. **基础设施层独立**：网络层、数据层、公共组件库、路由、状态管理应作为独立阶段，不混在功能模块里
2. **UI 文档分层**：全局 Token（色值/字体/间距/圆角）先行 + 模块级 UI 文档按需整理
3. **路径 B 缺业务逻辑来源**：Figma 只能给 UI 结构，业务逻辑需要从 PRD/API 文档/产品沟通中补全
4. **缺少验证对比环节**：每个模块实现完需要对比验证

### 第二轮：用户反馈与共识

用户对 4 点补充的回应：
1. ✅ 认同基础设施独立
2. ✅ 全局 Token 先行，但通用组件和页面级颜色在实现时按需处理（不预建）
3. ✅ Figma 路径需要额外的业务逻辑沟通环节
4. ✅ 需要验证步骤，但具体如何验证需要调研（特别是 Harmony 平台的自动化能力）

### 第三轮：Harmony 平台验证能力调研

通过 librarian + web search 调研，确认 Harmony 平台的验证工具链：

| 能力 | Android | Harmony |
|------|---------|---------|
| 设备连接 | adb | hdc |
| 截屏 | `adb shell screencap` | `hdc shell snapshot_display -f /data/local/tmp/test.jpeg` |
| 控件树 | `adb shell uiautomator dump` | `hdc shell uitest dumpLayout`（JSON 格式） |
| UI 自动化 | UIAutomator2 / Espresso | arkxtest（官方）/ hmdriver2（社区 Python 框架，425+ stars） |
| 稳定性测试 | Monkey | Wukong |
| UI 输入模拟 | `adb shell input` | `hdc shell uitest uiInput` |

**hmdriver2 详情**：
- 社区驱动的 Python UI 自动化框架（MIT 协议）
- 无侵入式（无需在设备端安装额外 app）
- API 风格对齐 Android 的 uiautomator2
- 支持元素定位（by text/id/type）、截屏、手势操作、应用管理
- `pip install hmdriver2`

**验证策略共识**：
- L1 轻量验证（每模块必做）：人工截屏对比
- L2 控件树对比（推荐）：dumpLayout JSON vs uiautomator XML
- L3 自动化回归（后期建设）：hmdriver2 脚本
- 前期不建议在验证工具链上投入太多，功能还在快速迭代，验证基线不稳定

### 第四轮：文档结构讨论

**初始方案**：合并两份工作流文档为一份

**参考 crossgate iOS 项目的规则文件后调整**：

查阅了 `/Users/apple/Desktop/Dw/Code/iOS/crossgate_ios/.cursor/rules/` 下的 4 个规则文件：
- `i18n-refactoring.mdc` — iOS 国际化改造流程
- `ui-implementation.mdc` — iOS UI 实现规范（1213 行，非常详细）
- `page-dev.mdc` — iOS 页面业务逻辑开发流程
- `ui.mdc` — UI 开发基础规则

从中借鉴了 4 个关键设计：

1. **Plan-Only / Implement 两阶段分离**：默认只做规划产出文档，明确授权后才动代码，包含工具/文件权限白名单。这跟"先文档后实现"一致，但更进一步——把约束写成硬性规则
2. **按任务类型拆分独立规则文件**：每个文件聚焦一种任务类型，按需加载（`alwaysApply: false`）
3. **模板文档引用机制**：每个步骤开始前必须查阅对应模板（Model-Template、ViewModel-Template、Mock-Template）
4. **验收标准检查项**：Plan 和 Implement 阶段各有独立 checklist，可直接作为 QA Agent 的审查依据

**最终方案**：不合并，而是按职责拆分到独立目录

### 第五轮：通用模板化讨论

用户提出的核心诉求：
1. **文档版本管理**：每次重构保留历史快照，可回溯对比
2. **书籍式目录结构**：总入口大纲 + 章节对应功能模块 + 子章节层次分明
3. **公共规则/模板独立于功能目录**
4. **整套目录结构能作为通用模板，拿来即用**

关键决策：
- 总入口用 `INDEX.md`（避免与项目根目录 README.md 混淆）
- 章节内部不用 README.md
- archive 按文件级快照（改哪个快照哪个），不做整目录快照
- 章节编号在开发计划中记录，具体目录等任务开始时再创建
- 00 固定为项目搭建，01 固定为基础设施
- guides/ 暂时跟项目走，不做全局共享（项目不多，工作流还需持续优化）
- 所有文档去除项目特定内容，抽象为通用模板
- `references/` 不硬编码 `android/`，按项目情况组织（路径 A 放源平台梳理文档，路径 B 放设计稿导出物）

## 三、最终产出

### 目录结构

```

├── INDEX.md                              ← 总入口：项目大纲 + 全局目录索引
├── guides/                               ← 公共规则与规范（可跨项目复用）
│   ├── workflows/
│   │   ├── 项目启动工作流.md              ← 路径A(源码复刻) / 路径B(设计稿起步)
│   │   └── 模块开发工作流.md              ← Plan → Review → Implement → QA 循环
│   ├── standards/                         ← 预留，按技术栈添加
│   └── templates/
│       ├── 源码梳理文档模板.md             ← 10份文档的结构模板
│       ├── 架构设计文档模板.md             ← 架构设计产出格式
│       ├── 模块UI实现文档模板.md           ← UI细节整理格式
│       ├── 模块实现方案模板.md             ← 文件清单+Todo格式
│       └── 模块知识库文档模板.md           ← 知识沉淀格式
├── chapters/                              ← 功能模块章节（按实现顺序编号）
│   └── 00-项目搭建/

    └── v1.0/                              ← 重构前的4份原始文件
```

### 各文件内容概要

**INDEX.md**：
- 项目信息表（应用名/平台/语言/IDE/构建系统/包管理/参考实现）
- 架构总览占位
- 功能模块索引表（章节 + 状态标记：⬜ 未开始 | 🔵 进行中 | ✅ 已完成）
- 公共规则索引（workflows / standards / templates 链接）
- 参考资料索引
- 版本历史表

**项目启动工作流.md**：
- §1 适用场景：路径 A（有参考实现）vs 路径 B（无参考实现）
- §2 路径 A 四步：A1 源项目全局梳理 → A2 目标平台架构设计 → A3 开发计划编排 → A4 全局设计 Token 提取
- §3 路径 B 五步：B1 设计稿获取与 Token 提取 → B2 页面与功能模块梳理 → B3 业务逻辑确认 → B4 技术选型与架构设计 → B5 开发计划编排
- §4 模块实现前的 UI 文档整理（两条路径共有，含路径 A/B 各自的整理方式 + 新增颜色处理规则 + 通用组件提取规则）
- §5 验证对比环节（L1/L2/L3 分层策略 + 跨平台注意事项 + 验证工具要求）
- §6 与模块开发工作流的衔接图
- §7 Agent 角色分配总览表
- §8 Mermaid 可视化流程图（全局总览 + 单模块生命周期）

**模块开发工作流.md**：
- §1 Agent 角色定义（主 Agent / Review Agent / Coding Agent / QA Agent）+ 协作关系图 + 信息流向
- §2 执行模式约束（Plan-Only 默认不动源码 / Implement 需明确授权）
- §3 流程总览（6 步循环）
- §4 步骤详解：
  - ① 生成模块实现方案（核心价值：文件清单让 Coding Agent 零探索成本）
  - ② Review 审查（5 个维度，最多 2 轮）
  - ③ 逐 Todo 实现（每个 Todo 后构建 + commit，偏差反馈机制）
  - ④ QA 审查（最多 2 轮，不通过升级用户）
  - ⑤ 验证对比（链接到启动工作流 §5）
  - ⑥ 知识库更新
- §5 文档体系（模块级文档结构 ui-plan / impl-plan / knowledge + 生命周期）
- §6 跨会话恢复路径（INDEX.md → impl-plan.md → git log 三重印证）
- §7 失败恢复策略（连续 3 次失败：停止 → 回退 → 记录 → 咨询/升级）
- §8 完成标准 checklist（6 项）

**源码梳理文档模板.md**：
- 使用说明 + 梳理顺序建议
- 10 份文档的完整结构模板，每份包含具体的表格和章节定义：
  1. 架构总览（项目信息/目录结构/分层/模块划分/依赖规则）
  2. 网络层（框架/请求封装/拦截器/错误处理）
  3. API 接口（接口总览/详情表/鉴权方式）
  4. 数据模型（模型总览/核心模型字段/枚举定义）
  5. 数据库（类型版本/表结构/索引/迁移策略）
  6. 业务流程（流程清单/详情含触发条件+步骤+异常分支+流程图）
  7. UI 页面（页面清单/跳转关系/导航结构）
  8. 三方 SDK（清单含状态标记 ✅⚠️❌🚫 / 详情含关键 API 和替代方案）
  9. 配置与常量（硬编码常量/配置项/Feature Flag）
  10. 安全与加密（Token 管理/加密方式/签名校验/敏感数据处理）
- 每份文档末尾都有"待确认事项"章节

**架构设计文档模板.md**：
- 项目概述表
- 架构总览（分层 + 目录结构树）
- 关键架构决策表（决策点/方案/理由）
- 功能范围（包含功能表 + 明确排除表）
- 技术选型（三方 SDK 映射表 + 概念映射表）
- 核心业务逻辑还原规则（用伪代码/流程描述，不写完整实现）
- 路由设计（URL 契约 + 拦截规则）
- 本地存储策略（数据库表 + 轻量存储 Key + 不缓存的数据说明）

**模块 UI 实现文档模板.md**：
- 页面清单表
- 每个页面 7 个维度的详细模板：
  1. 布局结构（树形图）
  2. 组件清单（类型标记：标准/自定义/通用）
  3. 样式细节（容器/文本/图片 分表，含具体属性列）
  4. 状态变化（条件 → 影响元素 → 变化内容）
  5. 交互行为（触发元素/事件类型/行为）
  6. 数据绑定（UI 元素/数据字段/格式化规则）
  7. 空状态与错误状态（5 种状态：加载中/空数据/网络错误/服务器错误/无权限）
- 新增设计 Token 记录表
- 通用组件候选记录表（含决策规则：当前模块内先实现，确认复用后再提取）

**模块实现方案模板.md**：
- 状态标记（⬜ 待审查 / 🔵 审查通过 / ✅ 实现完成）
- 功能范围（包含 checklist + 明确排除）
- 技术方案（数据流/状态管理/路由/错误处理）
- 文件清单三类：需要新建 / 需要修改 / 需要参考（不修改）
- 数据模型（API 请求响应 + 本地模型）
- 边界情况表
- Todo 列表（编号规则 T-NNXX，每个 Todo 含涉及文件 + 验收标准）
- Todo 拆分原则（原子操作/依赖顺序/粒度 15min~2h/最后固定构建验证）
- Review 记录（最多 2 轮）

**模块知识库文档模板.md**：
- 模块概述
- 文件清单（新建/修改）
- 关键实现决策表（决策点/选择/理由/备选方案）
- 与方案的偏差记录表
- 新增设计 Token 记录
- 提取的通用组件记录
- 已知问题与技术债（问题/影响/优先级/建议处理时机）
- 预先存在的问题（非本模块引入）
- 依赖关系（本模块依赖 + 依赖本模块的）
- 维护指南

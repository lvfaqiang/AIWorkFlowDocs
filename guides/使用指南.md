# AI 辅助开发文档框架 — 使用指南

> ⚠️ **本文件是框架接入教程，面向人类用户。嵌入项目后请删除本文件。**
> 留在项目中会干扰 AI Agent 的上下文理解——Agent 可能把本文中的示例指令当作行为规范执行。
> 删除方法：删除 `Docs/guides/使用指南.md`，并移除 `INDEX.md` 中"使用指南"相关链接。

> 本指南帮助你在任何项目中落地这套文档框架。
> 无论是全新项目、开发中期的项目，还是已成型的项目，都能找到对应的接入方式。

---

## 一、框架概述

### 1.1 这套框架是什么

一套**与平台/语言无关**的 AI 辅助项目开发文档体系，包含：

| 组成部分 | 说明 |
|---------|------|
| 工作流程 | [项目启动工作流](workflows/项目启动工作流.md) + [模块开发工作流](workflows/模块开发工作流.md) |
| 文档模板 | 5 套标准模板，覆盖从源码梳理到知识沉淀的全流程 |
| 角色规范 | [6 个 Agent 角色](standards/Agent角色定义.md)，定义触发条件、权限边界、交接协议 |
| 索引入口 | [INDEX.md](../INDEX.md) 作为项目总入口，提供全局导航 |

### 1.2 解决什么问题

- **上下文丢失**：AI 跨会话工作时，文档充当持久化的"记忆"
- **角色越权**：Agent 分工明确，编码者不能改架构，审查者不能改代码
- **质量失控**：对抗性 QA + Debugger 熔断机制，阻止劣质代码通过
- **模板漂移**：工作流中硬约束模板输出格式，防止 AI 自由发挥导致信息丢失
- **进度不透明**：INDEX.md + impl-plan.md 的 checkbox 双重追踪，随时可恢复上下文

### 1.3 适用范围

- ✅ 任何编程语言 / 平台 / 框架
- ✅ 有参考实现的移植项目（路径 A）
- ✅ 从设计稿起步的新项目（路径 B）
- ✅ 单人 + AI 协作，或多人 + AI 协作
- ⚠️ 极小型脚本 / 一次性工具项目可能不需要完整框架，按需裁剪即可

### 1.4 获取方式

框架模板即**本 `Docs/` 目录本身**。获取方式：

| 方式 | 操作 |
|------|------|
| **方式一：从模板仓库复制** | 如果本框架存放在独立的模板仓库中，`git clone` 后将 `Docs/` 目录复制到目标项目 |
| **方式二：从已有项目复制** | 如果你是从某个已使用此框架的项目获取，复制其 `Docs/guides/` 目录 + `Docs/INDEX.md` |
| **方式三：手动创建** | 按本指南 §二 的目录结构手动创建目录，然后逐个复制模板文件 |

> **核心原则**：你需要的是 `guides/` 下的全部文件（规则 + 流程 + 模板）和一份空白的 `INDEX.md`。`chapters/`、`references/` 是项目特定内容，应该是空的。

## 二、5 分钟快速开始

> 不想读完整指南？按这 3 步做完就能开始。

### ① 复制框架文件
```bash
# 将框架的 guides/ 和 INDEX.md 复制到你的项目
cp -r /path/to/框架源/Docs/guides/ /path/to/你的项目/Docs/guides/
cp /path/to/框架源/Docs/INDEX.md /path/to/你的项目/Docs/INDEX.md
mkdir -p /path/to/你的项目/Docs/chapters /path/to/你的项目/Docs/references
```

### ② 创建 AI 入口配置

在项目**根目录**创建 AI 工具的配置文件，内容如下：

| AI 工具 | 配置文件名 |
|---------|-----------|
| OpenCode | `AGENTS.md` |
| Cursor | `.cursorrules` |
| Windsurf | `.windsurfrules` |
| 其他 | 参考对应工具的文档 |

将以下内容复制到配置文件中（根据实际情况修改 `[ ]` 中的占位符）：

```markdown
# [项目名] 项目规则

## 文档体系

本项目使用 AI 辅助开发文档框架。

- 项目总入口：`Docs/INDEX.md`（每次新会话必读）
- 工作流程：`Docs/guides/workflows/`
- Agent 角色规范：`Docs/guides/standards/Agent角色定义.md`
- 文档模板：`Docs/guides/templates/`

## 首次启动任务

如果 `Docs/INDEX.md` 中的项目信息仍包含占位符（如"填写xxx"），请执行以下初始化：
1. 扫描项目源码，自动填写 INDEX.md 的项目信息、架构总览、功能模块索引
2. 删除 `Docs/guides/使用指南.md`（接入教程，已完成使命）
3. 移除 `INDEX.md` 中"使用指南"相关链接
4. 按 `Docs/guides/workflows/项目启动工作流.md` 继续

## 工作流程

1. 新会话开始 → 先读 `Docs/INDEX.md` 确认项目全貌和当前进度
2. 开始模块开发 → 按 `Docs/guides/workflows/模块开发工作流.md` 执行
3. 文档产出严格按 `Docs/guides/templates/` 中的模板格式
4. Agent 角色分工遵循 `Docs/guides/standards/Agent角色定义.md`

## 关键约束

- Plan 阶段只读不写源码
- Implement 阶段只修改方案文件清单中声明的文件
- 连续 2 次修复失败必须启动 Debugger 诊断流程
- QA 必须逐 AC、逐文件穷尽检查
```

### ③ 对 AI 说第一句话

```
请阅读项目配置和 Docs/INDEX.md，执行首次启动任务，然后按项目启动工作流开始。
我的项目是 [简要描述]，[有/没有] 参考实现。[如有，参考实现源码位于 [路径]]
```

Agent 会自动完成：填写 INDEX.md → 删除使用指南并移除相关链接 → 进入项目启动工作流。

> 💡 **后续使用**：首次启动后，每次新会话只需对 AI 说"请阅读 Docs/INDEX.md 确认进度，继续工作"即可。

## 三、目录结构说明

```
你的项目/
├── Docs/                          ← 文档根目录
│   ├── INDEX.md                   ← 项目总入口（必须）
│   ├── guides/                    ← 规则与模板（必须，直接复制，不改内容）
│   │   ├── standards/
│   │   │   └── Agent角色定义.md    ← Agent 角色规范
│   │   ├── workflows/
│   │   │   ├── 项目启动工作流.md   ← 项目级流程
│   │   │   └── 模块开发工作流.md   ← 模块级流程
│   │   └── templates/
│   │       ├── 源码梳理文档模板.md  ← 路径 A 专用
│   │       ├── 架构设计文档模板.md  ← 项目搭建阶段
│   │       ├── 模块UI实现文档模板.md ← 每个模块开始前
│   │       ├── 模块实现方案模板.md  ← 每个模块方案
│   │       └── 模块知识库文档模板.md ← 模块完成后沉淀
│   ├── chapters/                  ← 按模块组织的项目文档（初始为空，使用时创建）
│   ├── references/                ← 参考资料（初始为空，可选）
└── src/                           ← 你的源码
```

### 各目录职责

| 目录/文件 | 职责 | 何时创建 |
|-----------|------|---------|
| `INDEX.md` | 项目全局导航、模块状态追踪、版本历史 | 框架接入时 |
| `guides/` | 存放不随项目变化的规则、流程、模板 | 框架接入时（直接复制，**不修改内容**） |
| `chapters/` | 存放项目特定的文档产出（方案、知识库等） | 项目启动后按需创建子目录 |
| `references/` | 参考实现的梳理文档或设计稿导出物 | 路径 A/B 需要时创建 |

## 四、场景 A：全新项目

> 适用于：项目尚未开始，或刚创建了空仓库。

### Step 1：复制框架文件

```bash
# 方法一：整目录复制（推荐）
cp -r /path/to/框架源/Docs/guides/ 你的项目/Docs/guides/
cp /path/to/框架源/Docs/INDEX.md 你的项目/Docs/INDEX.md
mkdir -p 你的项目/Docs/chapters 你的项目/Docs/references
```

复制哪些、不复制哪些：

| 来源文件/目录 | 操作 | 说明 |
|-----------|:---:|------|
| `guides/` 整个目录 | ✅ 复制 | 规则、流程、模板，直接使用不改内容 |
| `INDEX.md` | ✅ 复制 | Agent 首次启动时会自动填写项目信息 |
| `chapters/` | ⚠️ 创建空目录 | **不要复制**来源项目的 chapters 内容，那是别人的项目数据 |
| `references/` | ⚠️ 创建空目录 | 同上 |

### Step 2：创建 AI 入口配置

按 [§二 快速开始 → 步骤②](#-创建-ai-入口配置) 创建项目级 AI 配置文件。

### Step 3：对 AI 说第一句话

根据你的情况选择路径：

```
有参考实现（其他平台已有成熟版本）？
  ├── 是 → 路径 A：源码复刻
  │         按「项目启动工作流 §二」执行
  │         A1 源码梳理 → A2 架构设计 → A3 开发计划 → A4 设计Token
  │
  └── 否 → 路径 B：设计稿起步
            按「项目启动工作流 §三」执行
            B1 设计稿Token → B2 页面梳理 → B3 业务确认 → B4 架构 → B5 计划
```

**对 AI 说**（复制即用）：

路径 A：
```
请阅读项目配置和 Docs/INDEX.md，执行首次启动任务。
本项目有参考实现，请按路径 A 开始执行。参考实现源码位于 [路径]。
```

路径 B：
```
请阅读项目配置和 Docs/INDEX.md，执行首次启动任务。
本项目没有参考实现，请按路径 B 开始执行。设计稿链接：[链接]。
```

Agent 会自动完成：填写 INDEX.md → 删除使用指南并移除相关链接 → 进入项目启动工作流。

### Step 4：进入模块循环

项目启动工作流完成后，按顺序进入模块开发：

```
对于开发计划中的每个模块：
  1. 整理模块 UI 文档（项目启动工作流 §四）
  2. 进入模块开发工作流：
     ① 生成实现方案 → ② Review → ③ 逐 Todo 实现 → ④ QA → ⑤ 验证 → ⑥ 知识库
  3. 更新 INDEX.md 模块状态
  4. 进入下一个模块
```

**每个模块开始时对 AI 说**：
```
请阅读 Docs/INDEX.md 确认当前进度，然后按 Docs/guides/workflows/模块开发工作流.md 开始 [模块名] 模块的开发。
```

### Step 5：持续维护

- 每个模块完成后更新 `INDEX.md` 状态标记（⬜ → 🔵 → ✅）
- 在 `INDEX.md` 版本历史中记录重点变更，通过 Git Tag 保存快照

## 五、场景 B：项目开发中期接入

> 适用于：项目已开发了一段时间，有部分代码但缺乏系统文档。

### 评估现状

在接入前，先回答以下问题：

| 问题 | 影响 |
|------|------|
| 项目架构是否已确定？ | 决定是否需要补写架构设计文档 |
| 已完成了哪些模块？ | 决定知识库回填的工作量 |
| 是否有参考实现？ | 决定是否需要源码梳理 |
| 测试/验证基建是否就绪？ | 决定是否需要补建验证工具链 |

### Step 1：复制框架文件

同场景 A 的 Step 1——复制 `guides/` + `INDEX.md`，创建空的 `chapters/`、`references/`。

### Step 2：创建 AI 入口配置

按 [§二 快速开始 → 步骤②](#-创建-ai-入口配置) 创建项目级 AI 配置文件。

中期项目需要在配置文件的"首次启动任务"中额外补充：

```markdown
## 首次启动任务

如果 `Docs/INDEX.md` 中的项目信息仍包含占位符（如"填写xxx"），请执行以下初始化：
1. 扫描项目源码，自动填写 INDEX.md 的项目信息、架构总览
2. 扫描已完成的模块，在功能模块索引中回填真实状态（⬜/🔵/✅）
3. 删除 `Docs/guides/使用指南.md`（接入教程，已完成使命）
4. 移除 `INDEX.md` 中"使用指南"相关链接
5. 按 `Docs/guides/workflows/项目启动工作流.md` 继续
```

### Step 3：对 AI 说第一句话

```
请阅读项目配置和 Docs/INDEX.md，执行首次启动任务。
项目已开发到中期，[简要描述当前进展]。
[有/没有] 参考实现。[如有，参考实现源码位于 [路径]]
```

Agent 会自动完成：填写 INDEX.md（含已有模块回填）→ 删除使用指南 → 评估当前状态。

### Step 4：补写架构设计文档（如缺失）

如果项目没有架构设计文档，让 Agent 辅助生成：

```
请阅读 Docs/guides/templates/架构设计文档模板.md，然后扫描项目源码，
按模板格式生成 chapters/00-项目搭建/技术选型与架构设计.md 草稿。
重点记录现有架构决策，不需要重新设计。
```

### Step 5：回填已完成模块的知识库

对于每个已完成的模块，让 Agent 辅助回填知识库：

```
请阅读 Docs/guides/templates/模块知识库文档模板.md，然后扫描 [模块源码目录] 的代码，
按模板格式生成 chapters/{NN}-{模块名}/knowledge.md 草稿。
我会审核确认后再定稿。
```

**最小回填原则**——不需要事后补写完整的 impl-plan.md，只需沉淀关键信息：
- 模块包含的文件清单
- 关键实现决策和原因
- 踩过的坑和解决方案
- 与其他模块的依赖关系

> **原则**：回填是为了让 AI 后续工作时有上下文，不是为了补形式。只记录对后续开发有价值的信息。

### Step 6：对当前进行中的模块，按标准流程继续

从当前模块开始，严格按 [模块开发工作流](workflows/模块开发工作流.md) 执行：

- 如果模块还没有 impl-plan.md → 从步骤 ① 开始
- 如果已经在写代码 → 补写一份轻量 impl-plan.md（至少包含文件清单和 Todo），然后继续步骤 ③

### Step 7：后续模块正常流程

所有后续新模块，完整走 [项目启动工作流 §四](workflows/项目启动工作流.md#四模块实现前的-ui-文档整理两条路径共有) + [模块开发工作流](workflows/模块开发工作流.md)。

## 六、场景 C：已成型项目接入

> 适用于：项目已基本完成或长期维护阶段，需要引入 AI 辅助进行新功能开发或重大重构。

### 接入策略：轻量级知识捕获

已成型项目不需要回填所有历史文档。核心目标是：**建立足够的上下文，让 AI 能理解现有代码并安全地做增量修改**。

### Step 1：复制框架文件

同样复制完整的 `guides/` 目录 + `INDEX.md`，创建空的 `chapters/`。

> ⚠️ **为什么不精简复制？** INDEX.md 中引用了 `guides/` 下的所有文件链接。只复制部分文件会导致导航链接 404。全量复制 `guides/` 的成本很低（都是文本文件），暂时用不到的模板留着不碍事，未来需要时无需重新获取。

实际使用中，你只需要关注这些文件（其余保留但暂不使用）：

| 文件 | 作用 |
|------|------|
| `guides/standards/Agent角色定义.md` | Agent 角色规范 |
| `guides/workflows/模块开发工作流.md` | 增量开发的核心流程 |
| `guides/templates/模块实现方案模板.md` | 新功能/重构的方案模板 |
| `guides/templates/模块知识库文档模板.md` | 知识沉淀 |

### Step 2：创建 AI 入口配置

按 [§二 快速开始 → 步骤②](#-创建-ai-入口配置) 创建项目级 AI 配置文件。

Agent 首次启动时会自动扫描项目代码，填写 INDEX.md（项目信息 + 所有模块标记为 ✅），并删除使用指南。

### Step 3：对 AI 说第一句话

```
请阅读项目配置和 Docs/INDEX.md，执行首次启动任务。
这是一个已成型的项目，我需要 [新增功能/重构xxx]。
```

### Step 4：为即将修改的模块创建知识库

只对**即将被修改的模块**创建知识库，不需要全量回填：

```
请阅读 Docs/guides/templates/模块知识库文档模板.md，然后扫描 [模块源码目录] 的代码，
按模板格式生成 chapters/{NN}-{模块名}/knowledge.md 草稿。
重点关注：文件结构、关键实现决策、模块间依赖关系、已知的坑。
```

### Step 5：新功能 / 重构按标准流程

任何新功能或重构任务：

1. 按 [模块开发工作流](workflows/模块开发工作流.md) 执行
2. 从步骤 ① 开始——写 impl-plan.md
3. 经过 Review → 实现 → QA → 知识库更新 的完整循环

### Step 6：渐进式扩展

随着更多模块被触及，逐步扩展文档覆盖：

```
第一次触及某模块 → 创建 knowledge.md
第二次修改该模块 → 补充知识库中的"踩坑记录"和"变更历史"
多次修改后 → 该模块拥有完整的上下文文档
```

不追求一次性全覆盖，让文档随着实际开发自然生长。

## 七、快速参考

### 场景对照表

| 维度 | 新项目 | 中期项目 | 成型项目 |
|------|-------|---------|---------|
| 框架复制 | 完整 guides/ + 空 INDEX.md | 完整 guides/ + 空 INDEX.md | 完整 guides/ + 空 INDEX.md |
| AGENTS.md | 创建项目级 Agent 入口配置 | 创建项目级 Agent 入口配置 | 创建项目级 Agent 入口配置 |
| INDEX.md | Agent 扫描源码自动填写 | Agent 自动填写 + 回填模块索引 | Agent 自动填写 + 列出所有模块 |
| 架构文档 | 按流程产出 | 补写（记录现状，可 AI 辅助） | 一段话概述即可 |
| 已有模块 | 无 | 回填知识库（最小化，可 AI 辅助） | 按需创建知识库（AI 辅助） |
| 当前模块 | 按标准流程 | 补 impl-plan → 继续 | 按标准流程 |
| 后续模块 | 按标准流程 | 按标准流程 | 按标准流程 |

### Agent 角色速查

| Agent | 何时出现 | 核心规则 |
|-------|---------|---------|
| 主 Agent | 贯穿全流程 | 只做规划和协调，不直接写业务代码 |
| Review Agent | Plan 阶段结束后 | 只读审查，不改代码，不通过直接打回 |
| Coding Agent | 拿到审批通过的方案后 | 只改方案中列出的文件，偏差即停 |
| QA Agent | 所有 Todo 完成后 | 对抗性审查，逐 AC / 逐文件穷尽检查 |
| Debugger Agent | 连续 2 次修复失败后 | 独立上下文诊断，只给建议不改代码 |
| Explorer Agent | 任何需要搜索代码时 | 只读，绝对禁止写入 |

### 模板使用时机速查

| 模板 | 使用时机 | 产出位置 |
|------|---------|---------|
| [源码梳理文档模板](templates/源码梳理文档模板.md) | 路径 A：梳理参考实现时 | `references/{platform}/` |
| [架构设计文档模板](templates/架构设计文档模板.md) | 项目搭建阶段 | `chapters/00-项目搭建/` |

| [模块实现方案模板](templates/模块实现方案模板.md) | 模块开发步骤 ① | `chapters/{NN}-{模块名}/impl-plan.md` |
| [模块知识库文档模板](templates/模块知识库文档模板.md) | 模块完成后 | `chapters/{NN}-{模块名}/knowledge.md` |

### 跨会话恢复路径

当 AI 在新的会话中继续工作时：

```
1. 读 INDEX.md → 了解项目全貌 + 找到当前模块
2. 读当前模块的 impl-plan.md → 找到当前 Todo 进度
3. 读 knowledge.md（如有）→ 了解已沉淀的上下文
4. 检查 git log → 确认代码实际状态
5. 继续工作
```

## 八、常见问题

### Q1：框架文件需要放入版本控制吗？

**推荐放入**。`Docs/` 目录应该和源码一起提交到 Git：
- `guides/` 下的规则和模板是团队共享的标准
- `chapters/` 下的文档是项目知识的一部分

### Q2：可以只用部分模板吗？

可以。最小可用集是：
- `INDEX.md`（必须）
- `模块开发工作流.md`（必须）
- `模块实现方案模板.md`（必须）
- `Agent角色定义.md`（强烈推荐）

但建议**始终复制完整的 `guides/` 目录**。原因：
- INDEX.md 中的链接引用了所有模板，缺少文件会导致链接 404
- 暂时不用的模板不影响使用，未来需要时无需重新获取
- 全量复制更简单，不容易出错

### Q3：AI 工具有要求吗？

框架本身不绑定特定 AI 工具。但设计时假设 AI 具备：
- 读写文件能力
- 搜索代码库能力
- 执行构建/测试命令能力
- 多角色切换能力（或通过 prompt 模拟）

常见兼容工具：Cursor、OpenCode、Windsurf、Aider、Claude Code 等。

### Q4：章节编号 00 和 01 有什么特殊含义？

| 编号 | 含义 | 说明 |
|------|------|------|
| `00` | 项目搭建 | **固定**。存放技术选型、架构设计、开发计划 |
| `01` | 基础设施 | **固定**。存放网络层、存储层、路由、设计系统、验证基建 |
| `02`+ | 业务模块 | 按依赖顺序递增，被依赖的模块编号更小 |

### Q5：多人协作时怎么用？

- 每个人负责不同模块，各自在 `chapters/{NN}-{模块名}/` 下工作
- 共享 `INDEX.md` 作为进度看板
- `guides/` 下的规则对所有人生效
- 冲突主要发生在 `INDEX.md`（模块状态更新），建议约定合并策略

### Q6：项目技术栈变更怎么办？

- 更新 `INDEX.md` 项目信息表
- 更新 `chapters/00-项目搭建/` 中的架构设计文档
- 在 `INDEX.md` 版本历史中记录变更
- `guides/` 下的流程和模板**不需要修改**（它们是通用的）

### Q7：如何给 AI 指示使用这套框架？

在项目根目录创建 `AGENTS.md`（或你的 AI 工具对应的配置文件），写入步骤②中的模板即可。详见 [§二 快速开始 → 步骤②](#-创建-ai-入口配置)。

核心原理：`AGENTS.md` 是 AI Agent 进入项目后第一个读取的文件，它将 Agent 引导至 `Docs/INDEX.md`，从而进入整个文档体系。

> 💡 如果你的 AI 工具不支持 `AGENTS.md`，将步骤②中的模板内容复制到对应的配置文件中（如 `.cursorrules`、`.aider.conf`、`rules.md` 等），效果一致。

## 九、版本适配

本使用指南基于文档框架 **v4.1** 编写。

主要变更（v4.0 → v4.1）：
- 快速开始从 4-5 步简化为 3 步
- INDEX.md 改由 Agent 扫描源码自动填写，不再需要用户手动填写
- 新增项目级 `AGENTS.md` 作为 Agent 入口（取代 Q7 中的内联配置）
- 使用指南定位为接入教程，嵌入项目后由 Agent 自动删除

框架版本变更时，请关注 [INDEX.md 版本历史](../INDEX.md#版本历史) 中的更新说明。

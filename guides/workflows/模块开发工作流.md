# 模块开发工作流

> 覆盖单个功能模块从"实现方案"到"代码交付"的完整循环。
> 前置条件：已完成 [项目启动工作流](项目启动工作流.md) 中的项目搭建 + 模块 UI 文档整理。

---

## 一、Agent 角色定义

> 各角色的触发条件、行为边界、交接协议详见 [Agent 角色定义](../standards/Agent角色定义.md)。

| Agent | 角色 | 职责 |
|-------|------|------|
| 主 Agent | 编排者 | 沟通需求、生成 Plan、生成模块方案、任务拆分、协调各 Agent、维护知识库 |
| Review Agent | 审查者 | 审查全局 Plan、审查模块实现方案（Plan 阶段） |
| Coding Agent | 实现者 | 按 Todo 写代码、构建验证、commit |
| QA Agent | 质检者 | 对抗性代码审查，对照方案验证完整性和正确性（Implement 阶段） |
| Debugger Agent | 诊断者 | Coding Agent 连续 2 次失败后介入，独立上下文诊断根因 |
| Explorer Agent | 探索者 | 代码库搜索、模式发现、上下文收集（贯穿全流程） |

### 协作关系

```
用户
  │
  ▼
主 Agent（Orchestrator）
  │           │              │              │             │
  │           ▼              ▼              ▼             ▼
  │     Review Agent    Coding Agent     QA Agent    Explorer Agent
  │     (审查方案)      (写代码)        (质检代码)    (搜索代码库)
  │           │              │              │
  │     ┌─────┘              │         ┌────┘
  │     │ 审查反馈            │ 偏差反馈  │ 质检结果
  │     ▼                    ▼         ▼
  └──── 主 Agent 决策 ◄──────┘    通过 → 知识库更新
                │                 不通过 → Coding Agent 修复（最多 2 轮）
                │
          连续 2 次失败
                │
                ▼
         Debugger Agent（诊断根因）→ Coding Agent 按建议修复
```

## 二、执行模式约束

借鉴 Plan-Only / Implement 两阶段分离模式：

### Plan-Only 阶段（默认执行）

- 输入：功能需求 + UI 文档（已在启动工作流中产出）
- 产出：模块实现方案（严格按 [模块实现方案模板](../templates/模块实现方案模板.md) 输出，所有章节必须保留）
- 约束：**不修改任何源码文件**，仅在 `chapters/{NN}-{模块名}/` 中创建文档
- 允许的操作：读取文件、搜索代码库（grep/explore）、查阅外部文档（librarian）
- 禁止的操作：编辑/创建源码文件、执行构建命令、修改配置文件

### Implement 阶段（需明确授权）

- 触发：用户明确说"开始实现""可以写代码"等
- 范围：按实现方案中的文件清单执行
- 约束：严格按方案执行，遇到偏差反馈主 Agent
- 允许修改的文件：仅限 impl-plan.md §3 文件清单中"需要新建"和"需要修改"的文件
- 超出文件清单的修改：必须先反馈主 Agent → 主 Agent 更新文件清单 → 获得授权后再修改

## 三、流程总览

```
① 生成模块实现方案
  │
  ▼
② Review Agent 审查方案（最多 2 轮）
  │
  ▼
③ Coding Agent 按 Todo 逐个实现
  │
  ▼
④ QA Agent 审查代码产出（最多 2 轮）
  │
  ▼
⑤ 验证对比（参见启动工作流 §5）
  │
  ▼
⑥ 知识库更新
  │
  ▼
⑦ 回到启动工作流"选取下一个模块"
```

## 四、步骤详解

### ① 生成模块实现方案

主 Agent 产出模块实现方案，**严格按 [模块实现方案模板](../templates/模块实现方案模板.md) 输出，模板中的所有章节必须保留，不得省略**。存放在 `chapters/{NN}-{模块名}/impl-plan.md`。

核心内容：
- 功能范围（对照 UI 文档和开发计划）
- 技术方案（路由、数据流、状态管理、错误处理）
- 文件清单（需要新建 / 需要修改 / 需要参考）
- 边界情况
- Todo 列表（原子化、有序）

**文件清单是核心价值**：Coding Agent 拿到方案后直接知道该读哪些文件、改哪些文件、参考哪些文件，零探索成本。

---

### ② Review Agent 审查方案

Review Agent 从以下维度审查：

- **模板完整性**：impl-plan.md 是否覆盖模板中的所有章节（功能范围、技术方案、文件清单、数据模型、边界情况、Todo 列表），缺章节直接打回
- **需求覆盖度**：功能范围是否完整（对照 UI 文档 + 开发计划）
- **技术可行性**：方案是否与项目技术栈兼容
- **文件清单准确性**：有没有遗漏、有没有多余
- **前置依赖**：依赖的模块是否已完成
- **Todo 粒度**：是否合理（太粗会遗漏，太细会拖慢）

**完整性 Checklist**（逐项确认）：

| 维度 | 检查项 | 通过条件 |
|------|--------|---------|
| 页面入参 | §二 或 ui-plan 中是否列出了所有 Intent extras / Fragment args | 所有入口参数已列出，多入口行为分支已标注 |
| 生命周期 | §二 生命周期钩子是否列出了 onResume/onPause 等非 UI 逻辑 | 已列出或标注"无" |
| 事件通信 | §二 事件通信是否列出了 EventBus/Broadcast/Observer | 已列出或标注"无" |
| 业务逻辑 | 是否有遗漏的数据持久化/状态同步/外部调用 | 对照源码方法清单无遗漏 |
| API 状态 | §四 API 是否标注了可用状态 | ✅/⚠️/❌ 状态已标注 |
| 需求确定性 | §一 是否标注了需求来源和不确定项 | 已标注，⚠️ 项有处理方案 |

最多 2 轮迭代，仍有重大分歧则升级给用户决策。

---

### ③ Coding Agent 按 Todo 逐个实现

**输入**：
- 当前 Todo 的描述
- 完整的模块实现方案（文件清单、技术方案、参考模式）
- 项目构建命令

**执行规则**：
- 按 Todo 顺序逐个实现，每次只做一个
- 每个 Todo 完成后执行构建命令确保编译通过
- 每个 Todo 完成后更新方案文档中的 checkbox
- 每个 Todo 完成后 git commit，message 格式：`feat(模块): T-xxx Todo标题`
- 遇到方案偏差 → 反馈主 Agent → 主 Agent 更新方案 → 继续

**TDD 推荐策略**：
如项目已配置测试框架，推荐按 TDD 红-绿-重构循环执行每个 Todo：
1. RED：先写一个会失败的测试，覆盖当前 Todo 的验收标准
2. GREEN：写最小实现使测试通过
3. REFACTOR：在测试保持绿色的前提下优化代码结构

非强制——由项目技术栈和测试基建决定是否启用。

**失败升级规则**：
Coding Agent 修复同一问题连续失败 **2 次**后：
1. 停止修改代码
2. 将失败上下文移交 Debugger Agent（详见 [Agent 角色定义](../standards/Agent角色定义.md#五debugger-agent)）
3. Debugger Agent 在独立上下文中诊断根因
4. 诊断完成后 Coding Agent 按建议修复

**commit message 示例**：
```
feat(auth): T-101 创建登录入参校验
feat(auth): T-102 实现登录 handler
fix(auth): T-104 边界情况处理与错误码
```

---

### ④ QA Agent 对抗性审查

模块所有 Todo 完成后，QA Agent 以**对抗性审查**姿态介入（详见 [Agent 角色定义 — QA Agent](../standards/Agent角色定义.md#四qa-agent)）。

**审查方式**：
- 逐条对照 impl-plan.md 中的验收标准（AC），**每条 AC 必须给出 PASS/FAIL + 证据**
- 逐文件检查变更清单中的每个文件，不允许跳过
- 如所有检查通过，须在审查报告中声明"**穷尽检查，未发现问题**"

**审查维度**：
- **功能完整性**：代码是否实现了方案中列出的所有功能点
- **边界情况**：impl-plan.md §5 列出的边界场景是否全部处理
- **代码一致性**：代码风格是否与"需要参考"文件一致
- **防御性检查**：有没有硬编码、遗漏的错误处理、不合理的逻辑
- **工程质量**：构建 / 测试 / lint 是否通过

**发现问题的报告格式**：
```
- [FAIL] AC-{编号}: {验收标准描述}
  - 文件: {file:line}
  - 问题: {具体描述}
  - 建议: {修复方向}
```

QA 不通过 → 反馈给 Coding Agent 修复 → 修复后再次 QA，最多 2 轮，仍不通过则升级给用户。

---

### ⑤ 验证对比

参见 [项目启动工作流 §5 验证基建要求](项目启动工作流.md#五验证基建要求)。

最低要求：L1 轻量验证（必须调用 `make verify-l1` 等统封命令截取关键页面并对比）。

---

### ⑥ 知识库更新

QA + 验证通过后，主 Agent 执行：

1. 将模块实现中的关键信息沉淀到知识库文档，**严格按 [模块知识库文档模板](../templates/模块知识库文档模板.md) 输出，模板中的所有章节必须保留，无内容时标注"无"**。存放在 `chapters/{NN}-{模块名}/knowledge.md`
2. 更新 `INDEX.md` 中的模块状态标记
3. 归档实现方案（impl-plan.md 中的 Todo 全部勾选即为归档状态）

## 五、文档体系

### 章节编号规则

`chapters/` 下的目录以两位数字编号 `{NN}-{模块名}` 命名，编号规则：

| 编号 | 用途 | 说明 |
|------|------|------|
| `00` | 项目搭建 | 固定。技术选型、架构设计、开发计划 |
| `01` | 基础设施 | 固定。网络层、存储层、路由、设计系统、验证基建 |
| `02`+ | 业务模块 | 按模块依赖顺序递增，被依赖的模块编号更小 |

### 模块级文档结构

```
chapters/{NN}-{模块名}/
├── ui-plan.md          ← 模块UI实现文档（启动工作流 §4 产出）
├── impl-plan.md        ← 模块实现方案（本工作流 ① 产出）
└── knowledge.md        ← 模块知识库（本工作流 ⑥ 产出）
```

### 文档生命周期

| 文档 | 创建时机 | 更新时机 | 归档条件 |
|------|---------|---------|---------|
| ui-plan.md | 模块开始前 | 实现过程中发现遗漏时 | 模块完成 |
| impl-plan.md | 步骤 ① | Todo 完成时勾选 checkbox | 全部 Todo 完成 |
| knowledge.md | 步骤 ⑥ | 后续维护时补充 | 长期维护 |

## 六、跨会话恢复路径

```
读 INDEX.md → 找到当前模块和状态
  → 读模块 impl-plan.md → 找到当前 Todo
    → 检查 git log 确认代码实际状态 → 继续工作
```

git commit 历史作为进度的"第二信源"，与文档中的 checkbox 互相印证，确保恢复准确。

---

## 七、失败恢复

### 当修复失败时

1. 修复根因，不修复症状
2. 每次修复后重新验证
3. 不做随机尝试

### 连续 2 次失败

1. **停止**所有编辑
2. **移交** Debugger Agent 在独立上下文中诊断根因（详见 [Agent 角色定义 — Debugger Agent](../standards/Agent角色定义.md#五debugger-agent)）
3. **回退**到最后已知可工作状态
4. **记录**尝试过的方案和失败原因
5. Debugger Agent 诊断完成后，Coding Agent 按建议修复
6. 若 Debugger 建议仍无法解决 → **咨询** Oracle 或升级给用户

**绝不**：留下破损代码、继续碰运气、删除失败的测试来"通过"

## 八、完成标准

一个模块视为完成，当且仅当：

- [ ] impl-plan.md 中所有 Todo 已勾选
- [ ] 构建通过（零错误）
- [ ] QA 对抗性审查通过（审查标准详见步骤④，须满足：逐 AC 验证均 PASS、逐文件检查无遗漏、声明"穷尽检查"）
- [ ] L1 验证对比通过（调用统一基建命令截图对比）
- [ ] knowledge.md 已沉淀（严格按模板，所有章节保留）
- [ ] INDEX.md 状态已更新

如果验证发现预先存在的问题（非本模块引入）：
- 不修复，记录在 knowledge.md 中
- 报告："完成。注意：发现 N 个预先存在的问题，与本模块无关。"
